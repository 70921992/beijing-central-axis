## 二、问题根源分析

### 2.1 侧导航浮窗失效

**问题**：点击 ☰ 按钮无响应，或面板展开后被地图层覆盖。

**根源**：
1. `#markers-layer`（z-index 层叠上下文根）撑满整个 `#map-container`，覆盖了左上角的侧导航按钮。
2. `#map-container` 未设置 `touch-action: none`，移动端浏览器可能在地图区域劫持手势，阻止按钮点击。
3. 侧导航展开后被详情弹窗的 z-index 压制（侧导航 z-index: 80 < 详情弹窗 z-index: 200）。

**证据代码**：
```css
/* 修复前 */
#map-container { flex: 1; position: relative; overflow: hidden; } /* 缺少 touch-action */
#markers-layer { position: absolute; ...; pointer-events: none; } /* 全屏覆盖 */

/* 修复后 */
#map-container { touch-action: none; -webkit-touch-action: none; }
#markers-layer { touch-action: none; } /* 阻止手势穿透但不阻止点击 */
.landmark-marker { pointer-events: auto; touch-action: manipulation; } /* 标记可独立响应 */
```

### 2.2 标记点位置偏移

**问题**：地图缩放后，标记点位置与地图地标不重合，随缩放程度偏移加剧。

**根源**：
1. `flyToLandmark()` 在每次打开详情时都被调用，将地图视角移走，导致标记坐标相对于视口产生视觉错位。
2. 标记元素的 CSS `transform: translate(-50%, -50%) scale(1/scale)` 逆缩放存在累积误差，高缩放级别下偏移明显。

**证据代码**：
```javascript
// 修复前：每次打开详情都触发地图飞行
function openDetail(index) {
    ...
    flyToLandmark(index);  // ← 这行导致每次打开详情地图都飞走
}

// 修复后：移除自动飞行，仅保留用户主动操作
function openDetail(index) {
    ...
    // flyToLandmark 调用已移除
}
```

### 2.3 弹窗详情无法正常显示

**问题**：点击标记后详情卡片无法弹出，或弹出后内容空白/布局错乱。

**根源**：
1. `flyToLandmark()` 动画持续 500ms，期间状态混乱，可能导致 `openDetail` 的 DOM 操作被覆盖。
2. 两次调用 `flyToLandmark(index)` 导致动画重叠，transform 覆盖。
3. 移动端 `overflow: hidden` 在 `#map-container` 上实现，但侧导航处于容器外，被意外裁剪。

### 2.4 页面加载缓慢

**问题**：首屏加载时间过长，尤其在 4G/移动网络下。

**根源及已采取措施**：

| 优化项 | 优化前 | 优化后 | 状态 |
|--------|--------|--------|------|
| 主地图 | 10.5 MB PNG | 1.5 MB WebP（-86%） | ✅ 已完成 |
| Gallery 图片 | 原始 JPG（部分 >1MB） | WebP（平均 -57%） | ✅ 已完成 |
| 项目总大小 | 80.1 MB | 16.3 MB（-80%） | ✅ 已完成 |
| 字体加载 | @import（阻塞） | link + display=swap（非阻塞） | ✅ 已完成 |
| DNS 预连接 | 无 | preconnect + dns-prefetch | ✅ 已完成 |
| 地图预加载 | 无 | rel=preload | ✅ 已完成 |

## 三、具体技术实现策略

### 3.1 触控事件分层（已实施）

**策略**：通过 `touch-action` CSS 属性精确控制每个层的触控行为。

```
#map-container           touch-action: none        地图区域手势归地图处理
  └─ #markers-layer     touch-action: none        标记层不劫持手势
       └─ .landmark-    pointer-events: auto      标记独立响应点击
          marker         touch-action: manipulation 允许浏览器点击优化
```

**效果**：侧导航按钮（位于 markers-layer 之外）不再被地图层遮挡。

### 3.2 详情弹窗触发逻辑重构（已实施）

**策略**：解耦"打开详情"与"地图飞行"两个独立操作。

- 用户点击标记 → 打开详情（不飞行）
- 用户点击侧导航项 → 关闭侧导航 + 飞行 + 打开详情
- 用户主动想要定位 → 使用独立的手动定位功能

**代码变更**：
```javascript
// openDetail() 中移除 flyToLandmark(index)
// 侧导航点击时：closeSideNav() → flyToLandmark(i) → openDetail(i)
// 标记点击时：closeSideNav() → openDetail(i)
```

### 3.3 标记点击自动关闭侧导航（已实施）

**原因**：侧导航展开时占据左侧部分区域，与地图标记重叠。用户点击标记时若侧导航未关闭，可能误触侧导航项。

```javascript
el.addEventListener('click', function(e) {
    closeSideNav();   // 避免侧导航误触
    openDetail(i);
});
```

### 3.4 移动端触控防误触

**策略**：给所有可点击元素添加 `touch-action: manipulation`。

该值特点：
- 允许浏览器优化点击响应（无 300ms 延迟）
- 阻止双击缩放等手势被该元素捕获
- 不阻止滚动、捏合缩放等手势

已应用元素：`#map-container`、`.landmark-marker`、`.gallery-stage`、`.side-nav-toggle`、所有 `.icon-btn`。

### 3.5 响应式尺寸体系

**策略**：针对移动端小屏特点，缩小各元素尺寸以适应 320px 宽度。

| 元素 | 桌面端 | 移动端（≤640px） |
|------|--------|------------------|
| 画廊高度 | 300px | 220px（最小 180px） |
| 标记点标签字号 | 13px | 11px |
| 悬浮信息标题 | 28px | 22px |
| 侧导航面板宽度 | 160px | 130px |
| 切换按钮尺寸 | 46px | 38px |
| 侧导航 z-index | 80 | 90（确保高于地图层） |

## 六、性能优化目标达成情况

### 6.1 资源体积优化

| 资源类型 | 优化前 | 优化后 | 节省比例 |
|----------|--------|--------|----------|
| 主地图 | 10.5 MB（PNG） | 1.5 MB（WebP） | **86%** |
| Gallery 图片（27张大图） | ~29 MB | ~9.4 MB（WebP） | **68%** |
| 项目总体积 | 80.1 MB | 16.3 MB | **80%** |

### 6.2 加载优化措施

| 措施 | 说明 |
|------|------|
| WebP 格式 | 相比 PNG/JPG 支持更优压缩，浏览器原生解码 |
| rel=preconnect | 提前建立 Google Fonts 连接，消除 DNS 解析等待 |
| rel=preload | 主地图 WebP 预加载，避免渲染阻塞 |
| display=swap | 字体加载期间显示回退字体，不阻塞首屏 |
| 图片懒加载 | 侧导航缩略图使用 `loading="lazy"` |

### 6.3 移动端内存优化

- 标记点使用 CSS `will-change: transform` 仅在需要时启用
- 画廊使用双幻灯片复用（仅 2 个 DOM 节点），而非创建 N 个图片元素
- `preloadImages` 设置 3 秒安全超时，避免无限等待
- `updateTransform` 动画结束后移除 `transition` 避免持续重绘

## 八、总结

本次改进方案通过以下核心技术手段系统性解决了移动端适配问题：

1. **触控分层**：`touch-action` 精确划分手势归属，地图层不再拦截按钮点击
2. **逻辑解耦**：移除详情打开时的自动地图飞行，避免 transform 动画冲突
3. **交互优化**：点击任何可交互元素前自动关闭侧导航，消除 z-index 遮挡
4. **资源压缩**：WebP 格式整体减少 80% 体积，16.3MB 总大小在 4G 下可接受
5. **响应式加固**：从 320px 到 1920px 全覆盖，CSS 断点清晰

所有 P0 紧急修复均已完成，项目在移动端的核心功能（查看地标、浏览详情、展开导航）现已可用。
